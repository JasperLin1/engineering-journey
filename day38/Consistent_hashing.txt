Consistent Hashing (⼀致性雜湊)
在分散式系統中，資料通常需要分散到多台伺服器上。
最直觀的⽅法是⽤ 簡單 hash： server = hash(key) % N   # N = 節點數

問題是：
如果 N 改變（新增或移除伺服器），幾乎所有資料都需要重新分配。
這會造成⼤量搬遷，導致 效能問題與系統不穩定

Consistent Hashing 的核⼼價值： 當伺服器數量改變時，只需要搬動⼀⼩部分資料，⼤部分資料仍然留在原來的節點。

Hash Ring
    把所有可能的 hash 值映射到⼀個圓環上（0 → MAX_HASH）。
    每個節點 (server) ⽤ hash 函數映射到圓環上的⼀個位置。
    每筆資料也⽤ hash(key) 計算位置。
    規則：資料順時針找到第⼀個節點，交給它存。

新節點加入「插隊」:
    假設圓環上有三台伺服器（A、B、C），資料會順時針找最近的伺服器。現在加入一台新伺服器 D。
    動作：D 選一個位置插進圓環（例如插在 A 和 B 之間）。
    受影響的人：只有原本要走去 B 的一部分資料，現在發現 D 離它們更近，所以改去 D。
    其他人：去 A 和 C 的資料完全不受影響，路徑一模一樣。

節點移除
    假設伺服器 B 壞掉了或是要關掉它。
    動作：B 離開圓環。
    受影響的人：原本要去 B 的資料，現在發現 B 不見了，於是繼續順時針走，直到遇到下一個伺服器 C。
    其他人：原本就去 A 或 C 的人，完全沒感覺，照樣工作。

虛擬節點 (Virtual Nodes)
    為什麼需要虛擬節點:
    在 Consistent Hashing 的基本版本裡：
    每個實體節點 (server) 在 hash 環上只有⼀個位置。
    問題：
        1. 資料分佈不均：hash 雖然隨機，但節點少時，分配可能很不平均。
        移除節點 A 以後，節點 Ｂ 需要負擔全部⼀半的 hashed values。
        2. 節點性能不同：有的伺服器性能強、有的弱，但⼀樣只拿到⼀個區間 → 沒有照硬體能⼒分配。
        *** 解法：引入虛擬節點。

虛擬節點的做法
    每個實體節點對應到多個「虛擬節點 (vNodes)」。
    vNodes 也會映射到 hash 環上，就像多個⼩節點。
    資料依舊依照 順時針規則 找到最近的 vNode 存放，然後交由對應的實體節點。

    這樣⼀來：A、B、C 各⾃擁有多個位置。 key 的分布更隨機、更均勻。

虛擬節點的好處
    1. 平均分佈：即使節點數少，也能把資料更均勻地切開。
    2. 依硬體能⼒分配：可以讓效能較強的機器放更多 vNodes，負責更多資料；效能較
    弱的機器放少⼀點。
    3. 彈性擴展：當節點加入/移除時，因為每個節點有多個 vNodes，搬遷的資料更細顆
    粒化，負載更平滑。

什麼時候要想到 Consistent Hashing？
*** 在以下情境，只要「節點數可能變動、要做 sharding/routing、希望最少搬遷」，就應優先考慮。 ***
