網路層次「OSI 模型」七層，在系統設計⾯試中最常出現的只有三個關鍵層次
    1.網路層（Network Layer） （Layer 3）: 這⼀層是 IP 協定，負責路由和定址。它把資料拆分成封包（packet）、處理封包在網路之間的轉送，把封包送往網路上任意的⽬的地 IP 位址
    2.傳輸層（Transport Layer ）Layer 4）:這⼀層有 TCP、QUIC 和 UDP，提供端到端的通訊服務。把它們想成在網路層之上增加了可靠性、排序、流量控制等功能的⼀層
    3.應用層（Application Layer）Layer 7）:最頂層是應⽤程式協定，例如 DNS、HTTP、WebSocket、WebRTC。這些常⾒的協定建立在 TCP（或 UDP，以 WebRTC 為例）之上，為通常和 Web 應⽤相關的不同類
                                            型資料提供抽象層。

Example：⼀個簡單的 Web 請求
⾸先，DNS 把⼈類可讀的域名（例如 https://www.buildmoat.org）轉換成 IP
位址（例如 32.42.52.62）。然後，⼀系列精⼼編排的步驟開始了：建立 TCP 連線、
發送 HTTP 請求、取得回應、拆除連線。

詳細流程：
1. DNS 解析：客⼾端⽤ DNS（Domain Name System）把網站的域名解析成 IP 位址。
2. TCP 三次握⼿（Three-Way Handshake）：
    SYN：客⼾端發送 SYN（同步）封包給伺服器，請求建立連線
    SYN-ACK：伺服器回應 SYN-ACK（同步確認）封包，確認請求
    ACK：客⼾端發送 ACK（確認）封包，完成連線建立
3. HTTP 請求：TCP 連線建立後，客⼾端向伺服器發送 HTTP GET 請求。
4. 伺服器處理：伺服器處理請求、取得網⾴、準備 HTTP 回應。（這通常是⼤多數軟體⼯程師唯⼀會想到並能控制的延遲！）
5. HTTP 回應：伺服器把包含網⾴內容的 HTTP 回應發回客⼾端。
6. TCP 四次揮⼿（Four-Way Teardown）：
    FIN：客⼾端發送 FIN（結束）封包，通知終⽌連線
    ACK：伺服器確認 FIN 封包
    FIN：伺服器發送 FIN 封包，終⽌它這端的連線
    ACK：客⼾端確認伺服器的 FIN 封包

傳輸層協定
傳輸層是建立應⽤程式之間端到端通訊的地⽅。這⼀層的三個主要協定是 TCP、UDP 和 QUIC，各⾃有不同的特性，適合不同的使⽤場景
QUIC 是
⼀個新的協定，⽬標是提供 TCP 的部分優勢並加入現代化的效能改進，目前相對還是新的協定，尚未廣泛普及。 把它視為更好版本的 TCP，只是沒有相同廣泛的基準⽀援

UDP (User Datagram Protocol) ：快速但不可靠，提供⼀個簡單的、無連線的服務，不保證送達、不保證順序、不保護重複。
    無連線：不需要握⼿或連線建立
    不保證送達：封包可能在沒有任何通知的情況下遺失
    不保證順序：封包可能以和發送不同的順序到達
    更低的延遲：較少的額外開銷意味著更快的傳輸

TCP (Transmission Control Protocol)：可靠但有額外開銷，是網際網路的主力。它提供可靠的、有序的、有錯誤檢查的資料交付。它透過三次握手建立連線。
這個連線叫做「串流（stream）」，是客⼾端和伺服器之間的有狀態連線⸺它也給了
我們討論順序的基礎：在同⼀個串流/連線中發送的兩則訊息，會以相同的順序到達。
TCP 會確保訊息的接收⽅確認收到，如果他們沒有確認，就會重新傳輸，直到被確認為⽌。
TCP 的關鍵特性：
    ⾯向連線：在資料傳輸前建立專⽤的連線
    可靠交付：保證資料有序且無錯誤地到達
    流量控制：防⽌傳送⽅以超過接收⽅處理能⼒的速率發送資料
    壅塞控制：適應網路壅塞，防⽌網路崩潰
TCP 非常適合資料完整性⾄關重要的應⽤⸺也就是幾乎所有 UDP 不適合的場景。

考慮使⽤ UDP 的情況：
低延遲⾄關重要（即時應⽤、遊戲）
⼀些資料遺失是可以接受的（媒體串流）
處理⾼流量的遙測或⽇誌，偶爾的遺失可以接受
不需要⽀援網⾴瀏覽器（或者有針對瀏覽器的替代⽅案）
