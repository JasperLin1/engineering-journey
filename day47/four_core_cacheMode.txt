Cache-Aside (旁載快取)
    最常⾒的快取模式，也就是⾯試中應該預設使⽤在Redis的那個。
    運作⽅式：
        Application 先查快取
        如果資料在，直接回傳
        如果不在，從資料庫取，存入快取，再回傳

    Cache-aside 只在需要時才快取資料，讓快取保持精簡。缺點是 cache miss 會造成額外的延遲。

Write-Through (同步寫穿)
    運作⽅式：
        application 只寫入快取
        快取再同步地把資料寫到資料庫
        整個寫入操作要等到快取和資料庫都更新完才算完成

    Redis 本⾝原⽣不⽀援 write-through，需要 application 程式碼或框架來實作這個模式。
    取捨在於寫入更慢，application 必須等待快取更新和資料庫寫入都完成。Write-
    through 也可能⽤那些再也不會被讀取的資料污染快取。


Write-Behind / Write-Back (非同步回寫)
    運作⽅式：
        pplication 只寫入快取
        快取在背景非同步地把資料批次寫到資料庫

    這讓寫入非常快，但引入了風險：如果快取在把資料 flush 到資料庫之前崩潰，資料就遺失了。
    適⽤場景：需要⾼寫入吞吐量，且可以接受最終⼀致性的⼯作負載。常⾒於分析和metrics pipeline。

Read-Through (讀穿快取)
    運作⽅式：
    不直接和資料庫溝通
    當cache miss 發⽣，是由快取本⾝去資料庫取資料、存起來
    再回傳給 application

    這是 write-through 的讀取端對應版本。在這兩種模式中，快取都充當處理資料庫操作
    的中間層，read-through 管理讀取，write-through 管理寫入。系統常常把兩者結合使⽤。


快取淘汰策略
    LRU（最近最少使⽤，Least Recently Used）：移除最久沒有被存取的項⽬。
        ⽤linked list 或 ring buffer 追蹤存取順序，可以在常數時間內移除最舊的項⽬。因為它對
        ⼤多數「最近⽤過的資料很可能再次被⽤」的⼯作負載都能良好適應，是許多系統的
        預設選擇。

    LFU（最不常使⽤，Least Frequently Used）：移除被存取次數最少的項⽬。為每個
        key 維護⼀個計數器，移除頻率最低的那個。部分實作使⽤近似 LFU 來避免精確頻率
        追蹤的成本。適合某些 key 長期持續熱⾨的場景，例如熱⾨影片或排⾏榜播放清單。

    FIFO（先進先出，First In First Out）：只根據插入時間，移除最早進入快取的項⽬。
        可以⽤簡單的 queue 實作，但它忽略了使⽤模式，可能移除還在被熱⽤的項⽬，
        所以除了簡單的快取層，在真實系統中很少使⽤。
        
    TTL（存活時間，Time To Live）：TTL 本⾝不是淘汰策略，⽽是為每個 key 設定過
        期時間，移除太舊的項⽬，常和 LRU 或 LFU 搭配使⽤，在資料新鮮度和記憶體⽤量之
        間取得平衡。只要資料必須最終刷新（例如 API 回應或 session token），TTL 就是必
        備的。
