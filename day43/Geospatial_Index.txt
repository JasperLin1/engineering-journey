⼀般 B-Tree / Hash Index 是針對⼀維的 key（例如整數 ID）。
但地理資訊是 ⼆維甚⾄多維 的（經度 longitude、緯度 latitude、⾼度 altitude…）。

常⾒的 Geospatial Index 技術
    Geohash
        Geohash 是⼀種將地球上的經緯度 (latitude, longitude) 壓縮成 ⼀維字串或整數 的⽅法。
        它把⼆維座標（經度、緯度）轉換成⼀個字串（通常是 Base32 編碼）。
        這個字串的「前綴」代表⼀個地理區塊。
        前綴越長，表⽰範圍越⼩、位置越精確。

        主要是想像成把整個世界先劃分成四個⼤⽅格，分別標記為 0–3。接著，再把每個⼤⽅格繼續
        劃分成更⼩的四個⽅格，如此不斷細分。每⼀次細分，都能讓地理位置描述更加精確。

        Geohash 基本上就是這個過程的數位化版本，只是它使⽤ Base32 編碼，把位置轉換
        成⼀串字串，例如 "dr5ru" 。字串越長，表⽰位置越精確：
        "dr" 可能代表整個舊⾦⼭
        "dr5" 會縮⼩到任務區 (Mission District)
        "dr5ru" 則可能精確到某⼀個城市街區
        Geohash 的巧妙之處在於：相近的地點通常會有相同的前綴字串。

        真正巧妙的地⽅在於：⼀旦我們把⼆維座標轉換成有序的 Geohash 字串，就能直接利
        ⽤現成的 B-tree 索引 來處理空間查詢。B-tree 擅長前綴比對與範圍查詢，這正好符
        合鄰近搜尋的需求。
        Geohash 受歡迎的原因之⼀：它簡單易懂、容易實作，並且能直接套⽤在現有的資料庫系統上。

    QuadTree(Quadrilateral Tree)
        Quadtree 是⼀種 空間劃分 (spatial partitioning) 的資料結構，主要⽤來處理 ⼆維空間資料。
        每個節點代表⼀個矩形區域 (region)。
        如果該區域的資料太多，就把它再分成 四個⼦區域 (quadrants)。
        這個過程可以遞迴進⾏，直到每個⼦區域的資料量在可接受範圍內。

        假設要找「舊⾦⼭某位置 5 公⾥內的餐廳」。
        在 Quadtree 中，整個城市地圖會先被當作⼀個⼤⽅格。當這個⽅格內的餐廳數量太多時，就會被切成四個象限：
        NE（東北）、NW（⻄北）、SE（東南）、SW（⻄南）。
        接著每個象限再根據資料量繼續切分，直到每個⽅格裡的餐廳數量⾜夠少，或已經到達允許的最⼩範圍。

        四分樹的主要優點是動態精度：資料密集的區域會被切分得更細，⽽資料稀疏的區域則維持較⼤的象限。
        四分樹需要專⾨設計的樹狀結構，也因為這種實作的複雜度。


    R-Tree (Rectangle Tree) Using MBR (Minimum Bounding Rectangle)
        R-tree 不像四分樹那樣把空間硬切成固定的四等份，⽽是利⽤ 彈性、可重疊的矩形 來組織資料。
        四分樹會不管資料怎麼分布，都⼀律把區域劃成四塊再繼續細分；相反地，
        R-tree 會根據實際資料的分布來調整矩形範圍。

        做法：直接根據數據的位置，把彼此接近的數據⾃然分成幾組，某一些組別可能會有⼀些重疊。
        R-tree 會先找到覆蓋整個城市的⼤矩形，然後逐層往下，透過⼀系列更⼩、彼此可能
        重疊的矩形，最後定位到單⼀餐廳的位置。這些矩形的⼤⼩與位置並不固定，⽽是會
        根據資料實際聚集的情況來調整。

        R-tree 可以在同⼀個索引結構中同時處理點與⼤型形狀。
        也就是說，⼀棵 R-tree 可以同時索引單⼀餐廳的位置、多邊形的外送區域，甚⾄
        道路網絡。矩形會⾃動調整⼤⼩。

        代價：
        矩形之間的重疊，有時會迫使查詢時要走訪多個分⽀。為
        了平衡這個問題，現代的 R-tree 實作通常會⽤更聰明的演算法，
        在「矩形重疊」與「樹深度」之間找到最佳平衡，並針對資料庫讀取磁碟的⽅式做優化。
        也正因如此，R-tree 成為現代⽣產環境中 空間索引的標準選擇。


        Google S2 Geometry / Uber H3
            S2 是 Google 設計的地理索引系統，把地球投影到⼀個立⽅體上，再把每個⾯切成更
            ⼩的四邊形 cell，每個 cell 都有唯⼀ ID。

            1. 把餐廳經緯度轉換成 S2 cell ID
                例如：台北⾞站 (25.0478, 121.5170) → S2CellId(… ) 。
                精度可以調整（cell 等級從 0 到 30，等級越⾼ cell 越⼩）。
            2. 把使⽤者的位置轉換成 S2 cell ID
                假設使⽤者在舊⾦⼭市中⼼ → 也會有對應 cell。
            3. 建立查詢區域 (S2RegionCoverer)
                設定搜尋半徑，例如 5 公⾥。
                S2 演算法會選出⼀組最少的 cell 來覆蓋這個圓形範圍。
            4. 查詢
                找出所有屬於這些 cell 的餐廳記錄。
                再⽤精確距離公式 (Haversine) 過濾。


        H3 是 Uber 開發的地理索引系統，核⼼概念是把地球投影到⼆⼗⾯體上，再切成六邊形 cell。
            把餐廳經緯度轉換成 H3 index
                例如： h3.geo_to_h3(lat, lng, resolution) 。
                resolution (解析度) 越⾼，cell 越⼩。
            把使⽤者位置轉換成 H3 index
                假設使⽤者在舊⾦⼭ Mission District → 會得到⼀個 H3 cell ID。
            找鄰近的 H3 cell
                ⽤ k_ring(cell, k) 找出以該 cell 為中⼼，半徑 k 的所有六邊形 cell。
                k 會依照半徑換算，例如要找 5 公⾥內的餐廳。
            查詢
                撈出屬於這些 H3 cell 的餐廳。
                同樣最後再做精確距離過濾。

        
        在同樣的「搜尋五公⾥內餐廳」的需求下，S2 會透過⽅格化的 cell 來覆蓋
        圓形範圍，⽽ H3 則會以中⼼ cell 為起點，擴展出⼀圈圈六邊形 cell 來逼近搜尋範
        圍。兩者都能有效率地縮⼩搜尋空間，但 S2 更適合⽤在地圖切片與範圍查詢，⽽ H3
        則在鄰近搜尋與分布式聚合上表現更好。
