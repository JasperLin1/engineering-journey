HTTP（Hypertext Transfer Protocol）是網路上資料通訊的事實標準。它是⼀個請求-回應協定，客⼾端向伺服器發送請求，伺服器回應所請求的資料。
    是⼀個無狀態協定，每個請求都是獨立的且伺服器不需要維護任何關於先前請求的資訊。
    ⼤多數簡單的 HTTP 伺服器可以被描述為請求參數的函式⸺它們是無狀態
    *請求⽅法：GET、POST、PUT、DELETE 等
    *狀態碼：200 OK、404 Not Found、500 Server Error 等
    *Headers：關於請求或回應的 metadata
    *Body：實際傳輸的內容
    GET ：從伺服器請求資料。GET 請求應該是冪等的，沒有 body。
    POST ：向伺服器發送資料。
    PUT ：更新伺服器上的資料。
    PATCH ：部分更新⼀個資源。
    DELETE ：從伺服器刪除資料。DELETE 請求應該是冪等的。

HTTPS(Hypertext Transfer Protocol Secure)
    在 HTTP 之上加了⼀個安全層（TLS/SSL）來加密通訊，防⽌竊聽和中間⼈攻擊。建立⼀個公開的網站，毫無例外都要⽤ HTTPS。
    雖然 HTTPS 的請求和回應內容是加密的，但這並不保證它們是由自己的客⼾端產⽣的！API 在沒有驗證的情況下，永遠不應該信任請求 body 的內容。
    ⼀個常⾒的錯誤是在請求 body 裡包含⽤⼾ ID，然後⽤它來做資料庫查詢。如果攻擊者能修改請求 body，他們就能修改⽤⼾ ID，讀取任意⽤⼾的資料。

雖然 HTTP 可以直接⽤來建立網站，但系統設計通常關注的是服務之間透過 API 的通訊。建立這些 API 有三個主要的範式：REST、GraphQL 和 gRPC。
REST(Representational State Transfer) ：簡單且靈活
    REST 是在系統設計⾯試中最常⽤的 API 範式。核⼼原則是客⼾端通常對「資源（resource）」執⾏簡單的操作（把資源想成資料庫表格或伺服器上的檔案）。
    RESTful API 設計利⽤ HTTP ⽅法加上關於路徑和請求 body 的⼀些慣例。通常⽤ JSON 表⽰請求和回應 body 中的資源。
    遵循系統設計的交付框架，核⼼實體（Core Entities）通常會直接對應到API 裡的資源。

    什麼時候⽤：REST 對各種使⽤場景都非常靈活。就像 TCP ⼀樣，REST 是⾯試的預設
    推薦選擇。除非有 REST 無法滿⾜的特定需求，才考慮 GraphQL、gRPC、SSE 或
    WebSocket。

GraphQL：彈性的資料獲取
    GraphQL 是⼀個較新的 API 範式，讓客⼾端能夠請求它精確需要的資料。
    前端和後端往往分屬不同的團隊。當前端團隊想顯⽰⼀個新⾴⾯，他們要嘛（a）拼湊⼀堆不同的後端端點請求（想像為了獲取 10 個⽤⼾的詳情
    ⽽發出 10 個 API 呼叫），要嘛（b）建立龐⼤的聚合 API（難以維護且難以變更），要嘛（c）為每個新⾴⾯寫全新的 API。這些都不是特別好的解法。
    GraphQL 解決了這些問題，讓前端團隊能靈活地向後端查詢它精確需要的資料，後端則以前端需要的形狀回應資料。

    什麼時候⽤：GraphQL 非常適合前端團隊需要快速迭代、靈活調整的使⽤場景，以及多個團隊對重疊資料進⾏廣泛查詢的情境。

    在系統設計⾯試中，GraphQL 的好處其實比較模糊。⾯試裡的需求是固定的（不像在迭代⾏動 App 或 Web 前端時那樣會移動），
    ⽽且 GraphQL 的後端執⾏有時會引入它本來試圖避免的那種複雜後端程式碼。主要只在問題明顯聚焦於靈活性時才提出GraphQL。

gRPC：強調效能與低延遲的通訊協定
    gRPC 是 Google 推出的⾼效能 RPC（遠端程序呼叫）框架，使⽤ HTTP/2 和 Protocol Buffers。
    Protocol Buffers 就像 JSON，但有更嚴格的綱要，帶來更好的效能和更有效率的序列化。
    不像帶有嵌入式綱要的 JSON（40 bytes，有了相同資料的⼆進位編碼（15 bytes），使⽤非常精簡的標籤和可變長度字
    串編碼。佔⽤空間更少，解析所需的 CPU也更少!

    這些定義被編譯成客⼾端和伺服器 stub，多種語⾔和框架都可以⽤來建構服務和客⼾端。gRPC 還包含許多對⼤規模運營微服務架構有⽤的功能，
    例如串流、截⽌時間、客⼾端負載平衡等。但最重要的是：它是⼀個比 HTTPover JSON 更快更有效率的⼆進位協定。

    什麼時候⽤：gRPC 在微服務架構中的服務間⾼效通訊上表現出⾊。強型別有助於在編譯時⽽不是執⾏時發現錯誤，
    ⼆進位協定比 JSON over HTTP 更有效率（某些基準測試顯⽰吞吐量⾼達 10 倍！）
