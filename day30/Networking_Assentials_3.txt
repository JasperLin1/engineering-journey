SSE（Server-Sent Events）：即時伺服器推送,單向
    SSE 是建立在 HTTP 之上的規範，讓伺服器能透過單⼀ HTTP 連線向客⼾端推送多則訊息
    理解 SSE 的⽅式是：它是 HTTP 之上的⼀個巧妙改裝，讓伺服器能在單⼀回應中、隨時間持續串流多則訊息
    特別之處,伺服器可以在單⼀回應中以「區塊」的形式推送多則訊息
    每⼀⾏作為⼀則獨立的訊息被接收,客⼾端可以在它到達時立即處理
    有⼀些固有的限制：SSE 連線不能保持太長時間，因為伺服器（或負載平衡器、中間代理）會關閉連線

    什麼時候⽤：在系統設計⾯試中，當你希望客⼾端⼀有通知或事件就立刻收到時，SSE
    是個好選擇。例如讓出價者即時看到拍賣當前價格

WebSocket：即時雙向通訊,雙向
    WebSocket 提供了客⼾端和伺服器之間持久的、類似 TCP 的連線，實現即時的雙向通訊，並有廣泛的⽀援（包括瀏覽器）
    不像 HTTP 的請求-回應模型，WebSocket 讓伺服器能在沒有新請求觸發的情況下主動推送資料給客⼾端，客⼾端也能隨時推資料給伺服器
    WebSocket 透過 HTTP「升級（upgrade）」協定發起，讓現有的 TCP 連線切換 L7協定。這非常⽅便，因為你可以利⽤現有的 HTTP ⼯作階段資訊
    (例如 cookies、headers 等)

    運作⽅式：
    1. 客⼾端透過 HTTP 發起 WebSocket 握⼿（底層是 TCP 連線）
    2. 連線升級到 WebSocket 協定，WebSocket 接管 TCP 連線
    3. 客⼾端和伺服器都可以透過連線互相發送⼆進位訊息
    4. 連線保持開啟，直到明確關閉為止

WebRTC：點對點通訊
    WebRTC 讓瀏覽器之間能夠直接進⾏點對點（peer-to-peer）通訊，不需要中間伺服器來做資料交換。
    WebRTC 非常適合視訊/⾳訊通話和會議應⽤，也可以⽤於協作應⽤（如⽂件編輯器）
    透過 WebRTC，客⼾端和⼀個中央「信令伺服器（signaling server）」溝通，信令伺服器追蹤哪些 peer 可⽤以及它們的連線資訊。
    ⼀旦客⼾端有了另⼀個peer 的連線資訊，就可以嘗試建立直接連線，不需要透過任何中間伺服器

    建立 WebRTC 連線的四個步驟:
    1. 客⼾端連線到中央信令伺服器，了解⾃⼰的 peer
    2. 客⼾端聯繫 STUN 伺服器取得⾃⼰的公網 IP 和 port
    3. 客⼾端透過信令伺服器與對⽅分享這些資訊
    4. 客⼾端建立直接的點對點連線，開始傳送資料

    什麼時候⽤：WebRTC 非常適合⾳視訊通話和會議應⽤。 在⾯試中，建議只在視訊/⾳訊通話和會議場景使⽤ WebRTC
